# Claude Code Prompt 1 of 21 — Merger plan: read both sites, produce the route map and collision audit

**Sequence:** prompt 3 of 22 (Wave 1 opener — the unification of theseus-public into theseus-codex). Every later prompt assumes the plan this one produces. Report back with: the produced `MERGER_PLAN.md`, the collision matrix, and any path deferred from automated migration.

**Why this prompt exists.** The repo currently has two Next.js projects: `theseus-codex/` (the founder portal, deploys to theseuscodex.com / mqtheseuswork-qiw6.vercel.app, with `/` already a public blog index and `/dashboard` already auth-gated) and `theseus-public/` (a separate static-export site with the firm's published research pages — methodology, conclusions, methods, interop, etc.). The user wants one site under theseus-codex. The Wave 1 merger does that move. Wave 2+ then adds the live-news (currents) feature on top of the merged app.

The merger is mostly content migration, not auth restructuring (the auth boundary is already correct in theseus-codex). But theseus-public has accumulated its own conventions, data sourcing, and design tokens, and dropping its files into theseus-codex's `src/app/` blindly will produce route collisions and broken imports. This prompt produces the plan that prompt 04 executes.

## Your task

Produce one document, `Claude_Code_Prompts/MERGER_PLAN.md`, with these sections:

### 1. Route map (current → target)

A table with three columns: source path in `theseus-public/src/app/`, target path in `theseus-codex/src/app/`, and a status column (`MIGRATE | COLLIDES | DATA_SOURCE_CHANGE | DEFER | UNCHECKED`).

Example rows:

| Source | Target | Status |
|---|---|---|
| `theseus-public/src/app/page.tsx` | `theseus-codex/src/app/page.tsx` | COLLIDES (theseus-codex already has its own homepage) |
| `theseus-public/src/app/methodology/page.tsx` | `theseus-codex/src/app/methodology/page.tsx` | MIGRATE |
| `theseus-public/src/app/c/[slug]/page.tsx` | `theseus-codex/src/app/c/[slug]/page.tsx` | DATA_SOURCE_CHANGE (was bundle.json, now Prisma) |

For every COLLIDES row, propose a resolution. The default for the homepage is: keep theseus-codex's `/` (it's already public + already aware of auth), and discard theseus-public's homepage. Currents teaser will be added in Wave 6.

### 2. Library + component map

For each top-level module in `theseus-public/src/lib/` and `theseus-public/src/components/`, decide:
- **REUSE-AS-IS** (file moves into theseus-codex unchanged, e.g. small utility helpers).
- **PORT** (file's logic is needed but the data source must change — Prisma, not bundle).
- **DROP** (functionality already exists in theseus-codex; remove from migration set).
- **DEFER** (uncertain; flag for the operator).

### 3. Asset migration

List every static asset (sculptures, OG images, fonts) under `theseus-public/public/` that does or does not exist in `theseus-codex/public/`. Resolve duplicates by content hash, not name.

### 4. Design-token reconciliation

For every CSS variable defined in either `globals.css`, list:
- Where defined.
- Whether the value matches across both sites.
- Resolution: keep theseus-codex's value, keep theseus-public's value, or RENAME one to avoid silent override.

If theseus-codex defines `--border: #e5e5e5` and theseus-public defines `--border: #d4d0c9`, that is a real visual regression risk. Pick the one the merged site should use. Document the decision.

### 5. Data sourcing decisions

For every theseus-public page that read from `bundle.json` (computed by `scripts/write-feeds.mjs`), describe how the merged page will source its data:

- For Conclusions list/detail: use `db.conclusion.findMany` with `where: { publishedAt: { not: null } }`.
- For Methods list/detail: use `db.method.findMany`.
- For Interop list/detail: use `db.interop.findMany`.
- For RSS/Atom feeds: keep the script-based generation but read from Prisma; produce the feeds at build time AND/OR at request time, depending on freshness needs.

Make these decisions concrete in the plan; prompt 04 will implement the safe subset.

### 6. Build + deploy implications

- Drop `vercel.json`'s `outputDirectory: "out"` setting that lives in theseus-public — the merged theseus-codex is a runtime Next.js app, not a static export. Confirm theseus-codex's existing `vercel.json` (if any) is correct.
- Confirm theseus-codex's `package.json` build command produces a working server build.
- Note that the prebuild step `node scripts/write-feeds.mjs` from theseus-public must be ported into theseus-codex's prebuild OR replaced with runtime feed generation — the merger plan must commit to one.

### 7. Risk register

List every risk that could land bad results:
- "If theseus-codex's `/c/[slug]/page.tsx` already exists with different params, the merge will overwrite it." — confirm or refute by reading.
- "If theseus-public's globals.css imports a font theseus-codex doesn't have, the merged app will 404 the font." — confirm or refute.
- "If a Conclusion has `publishedAt IS NOT NULL` but the bundle.json never knew about it, a fresh Prisma read will show MORE content than theseus-public did." — that may be fine, but confirm with the operator.

### 8. Migration order

The actual sequence prompt 04 will follow:

1. Move pure-content pages first (methodology, methods, interop) — least risk.
2. Move `/c/[slug]` and `/c/[slug]/v/[version]` next, with the data-source change.
3. Move `/conclusions/[id]`, `/open-questions`, `/responses`.
4. Update or rebuild RSS/Atom generation.
5. Migrate static assets and reconcile globals.css.
6. Update root layout's nav to include the migrated routes.
7. Verify nothing under `/dashboard` was touched.
8. Archive `theseus-public/` to `reference/theseus-public-pre-merge/`.
