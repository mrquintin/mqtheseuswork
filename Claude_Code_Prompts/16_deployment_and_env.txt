### 3. `Dockerfile.api`

```dockerfile
FROM python:3.12-slim
WORKDIR /app
ENV PYTHONDONTWRITEBYTECODE=1 PYTHONUNBUFFERED=1
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY noosphere ./noosphere
COPY current_events_api ./current_events_api
EXPOSE 8088
# One worker — OPINION_BUS is in-memory. Scale horizontally only with a real pub/sub.
CMD ["uvicorn", "current_events_api.app:app", "--host", "0.0.0.0", "--port", "8088", "--workers", "1"]
```

**Single-worker invariant:** the in-process `OpinionBus` fans out within one process. Multiple workers would each tail the DB independently and only serve the subset of clients bound to their worker — that's fine for at-least-once delivery but doubles token work. If the API ever needs horizontal scale, move the tailer + bus behind a real pub/sub; the current design does not support it.

### 7. Health + readiness endpoints

Create `ops/healthz.py`:

```python
from __future__ import annotations
import os, time, json
from pathlib import Path

def data_dir_ok() -> tuple[bool, str]:
    p = os.environ.get("NOOSPHERE_DATA_DIR")
    if not p: return False, "NOOSPHERE_DATA_DIR unset"
    if not Path(p).is_dir(): return False, f"{p} not a directory"
    return True, "ok"

def scheduler_liveness(max_age_seconds: int = 600) -> tuple[bool, str]:
    p = Path(os.environ["NOOSPHERE_DATA_DIR"]) / "currents_status.json"
    if not p.exists(): return False, "no status file yet"
    age = time.time() - p.stat().st_mtime
    if age > max_age_seconds: return False, f"status {int(age)}s old"
    try:
        last = json.loads(p.read_text())["last_cycle"]
    except Exception as e:
        return False, f"status unreadable: {e}"
    return True, f"last cycle {last.get('cycle_id')} {int(age)}s ago"
```

Modify `current_events_api/app.py`:

```python
from fastapi.responses import JSONResponse
from ops.healthz import data_dir_ok, scheduler_liveness

@app.get("/healthz", include_in_schema=False)
async def healthz():
    return {"status": "ok"}

@app.get("/readyz", include_in_schema=False)
async def readyz():
    checks = {
        "data_dir": data_dir_ok(),
        "scheduler": scheduler_liveness(),
    }
    status = all(ok for ok, _ in checks.values())
    return JSONResponse(
        status_code=200 if status else 503,
        content={"ok": status, "checks": {k: {"ok": v[0], "detail": v[1]} for k, v in checks.items()}},
    )
```

`/readyz` returns 503 if the scheduler has fallen silent — this is the right signal for a load balancer to drain traffic when the ingest pipeline is wedged.

### 8. Migrations

The `currents-migrate` init container runs `alembic upgrade head` and exits. Compose gates both `currents-api` and `currents-scheduler` on `service_completed_successfully` so neither starts against an unmigrated DB.

If this is the first deploy and the existing Noosphere SQLite file lives outside the new volume, document (in README) a one-time copy:

```bash
docker compose run --rm -v "$(pwd)/noosphere_data:/src:ro" currents-migrate \
  sh -c "cp -r /src/* /data/noosphere/ && alembic upgrade head"
```

```markdown
## Deploying

Two supported modes:

1. **All-in-containers** — `docker-compose.yml` as-is, run behind a reverse proxy
   that terminates TLS and forwards `:3001` and `:8088`. Single host is enough
   for the expected traffic; SQLite is authoritative and backed up via a volume
   snapshot.

2. **Hybrid (Vercel + host)** — deploy `theseus-public` to Vercel, the API and
   scheduler to a small VM. See `ops/deploy-vercel.md` for the exact Vercel
   configuration and the `CURRENTS_API_URL` that Vercel needs to reach your API.
```

```markdown
# Deploying theseus-public on Vercel with an external currents API

Since we dropped `output: "export"` in prompt 08, theseus-public now requires a
runtime. Vercel is a native target.

## Required environment variables (set in Vercel project settings)

- `CURRENTS_API_URL` — the publicly reachable URL of `current_events_api`
  (e.g. `https://currents-api.theseus.example`). TLS is mandatory.
- `PUBLIC_SITE_ORIGIN` — `https://theseus.example` (used for OG URLs).

## Routing

The Next.js route handlers under `/api/currents/*` proxy to `CURRENTS_API_URL`,
including the SSE streams. Vercel's Node runtime keeps long-lived connections
open under the Fluid pricing tier — confirm your plan supports it, or consider
serving the SSE endpoint directly from the API host and pointing `EventSource`
at that hostname instead. (That's a larger change and NOT in scope here.)

## Caching

`GET /api/currents/feed` uses `export const dynamic = "force-dynamic"` so Vercel
will not cache opinion lists. Do not add `revalidate` — live data must be live.

## Build

No special build step. `npm run build && npm run start` as usual. Do NOT set
`NEXT_BUILD_TARGET=standalone` on Vercel — standalone output is only for the
containerized mode.
```

## Prohibitions

- Do NOT introduce any external data store (Redis, Postgres, Kafka, S3). The entire system runs on SQLite on a mounted volume.
- Do NOT scale `currents-api` beyond `--workers 1` without first replacing the in-process `OpinionBus`. Multiple workers = split-brain SSE fan-out.
- Do NOT bake secrets into images or commit `.env`. `.env.example` is committed; `.env` must be in `.gitignore`.
- Do NOT expose `/metrics` or `/readyz` to the public internet without auth; they leak cardinality and internal state. Behind the reverse proxy, restrict both to internal IPs.
- Do NOT skip the `currents-migrate` gate. Starting `currents-api` against an unmigrated DB will 500 every request.
- Do NOT set `revalidate` or ISR flags on `/api/currents/*`. Live feed is live.

Do not proceed to any other work. Do not edit anything outside SCOPE.
