## The bug

Two server actions make HTTP requests back to their own server:

```typescript
// peer-review/[conclusionId]/page.tsx
async function runReview() {
  "use server";
  const base = process.env.PORTAL_API_BASE || "http://localhost:3000";
  const res = await fetch(`${base}/api/round3/review/run`, { ... });
}

// decay/page.tsx
async function revalidate() {
  "use server";
  const base = process.env.PORTAL_API_BASE || "http://localhost:3000";
  const res = await fetch(`${base}/api/round3/decay/revalidate`, { ... });
}
```

This has three problems:
1. **Auth cookies are not forwarded.** The server action runs on the server, so the browser's session cookie is not attached to the outgoing fetch. The API route calls `getFounder()` (via `withGated`), which reads cookies — so this likely returns 401 in production.
2. **Requires `PORTAL_API_BASE` env var.** The default `http://localhost:3000` breaks in any deployment where the server listens on a different port, hostname, or behind a reverse proxy.
3. **Unnecessary round-trip.** A server action already runs on the server. It should call the underlying logic directly instead of HTTP-fetching itself.

## Your task

Replace the self-fetching pattern with direct function calls. The server action should call the same underlying functions that the API route calls — `submitToRigorGate` and `callNoosphereJson` — without going through HTTP.
