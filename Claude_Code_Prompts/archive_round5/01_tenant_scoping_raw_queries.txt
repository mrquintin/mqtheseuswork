## The bug

Every raw SQL query in `round3.ts` — provenance, cascade, peer_review, decay_record, post_mortem, eval_run, rigor_gate_submission, method_registry, method_version, method_candidate — lacks an `organizationId` filter. In a multi-tenant deployment, any authenticated user can see data from any organization. The Prisma-backed queries (e.g. `db.conclusion.findMany` in the conclusions list page) are correctly scoped; only the raw SQL queries are unscoped.

## Your task

Add tenant scoping to every raw SQL read function in `round3.ts`. Each function should accept an `organizationId` parameter, and every `SELECT` must include `WHERE organization_id = ${organizationId}` (or `AND organization_id = ${organizationId}` if there's already a WHERE clause).

Then update every call site to pass the resolved organizationId from the session context.

## Step-by-step

1. **In `round3.ts`**, update every `fetch*` function signature to accept `organizationId: string` as its first parameter (or add it if the function currently takes no parameters). For functions that already take a specific ID (like `fetchProvenanceForConclusion(conclusionId)`), add `organizationId` as the first parameter.

2. **In every SQL query**, add `organization_id = ${organizationId}` to the WHERE clause. For queries that have no WHERE clause (like `fetchProvenanceRecords`), add one. For queries that already have a WHERE (like `fetchProvenanceForConclusion`), add an AND. Example:
   ```sql

   SELECT ... FROM provenance WHERE conclusion_id = ${conclusionId}
   -- After
   SELECT ... FROM provenance WHERE organization_id = ${organizationId} AND conclusion_id = ${conclusionId}
   ```

   ```typescript

   // inside the component:

   // then pass tenant.organizationId to the fetch function
   ```
