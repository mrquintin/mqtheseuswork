## Step-by-step

### 1. Interface — keep it stable

`EventRetrievalHit` stays exactly as prompt 03 declared it (`source_kind`, `source_id`, `text`, `score`). Add two optional fields that do not break the prompt-03 contract:

```python
@dataclass(frozen=True)
class EventRetrievalHit:
    source_kind: str         # "conclusion" or "claim"
    source_id: str
    text: str                # display text — for conclusions, the conclusion_text; for claims, .text
    score: float             # 0..1, comparable within one call
    topic_hint: Optional[str] = None
    origin: Optional[str] = None   # ClaimOrigin value when source_kind=="claim"; None for conclusions
```

### 2. Retrieval strategy

```python
def retrieve_for_event(
    store: Store,
    event: CurrentEvent,
    *,
    k_conclusions: int = 6,
    k_claims: int = 10,
    min_score: float = 0.25,
) -> list[EventRetrievalHit]:
    """
    Two-stage retrieval:

    Stage A (preferred): search Conclusion texts directly. A Conclusion is a
    firm-level published belief and is the strongest citation surface.

    Stage B (fallback / complement): run HybridRetriever over Claims filtered
    to ClaimOrigin in {FOUNDER, VOICE, LITERATURE} for the tenant, then re-rank.

    Merge, deduplicate by semantic overlap (a Claim already subsumed by a
    returned Conclusion should not be double-cited), sort by score descending,
    return up to (k_conclusions + k_claims) hits.
    """
```

Implementation steps:

1. **Build the query text** from the event: concatenate `event.raw_text` with `event.topic_hint` if set. Trim to 1500 chars.
2. **Stage A — Conclusions:**
   - Fetch all Conclusion rows (or the most recent N, e.g., 500 — tune per repo size). If the repo has a dense index over conclusions, use it; otherwise embed the query and score cosine against each Conclusion's embedding (Conclusions have embeddings in this repo — confirm by reading `noosphere/conclusions.py`).
   - Take top-k_conclusions above `min_score`.
3. **Stage B — Claims via `HybridRetriever`:**
   - Instantiate `HybridRetriever()`, call `.bm25_hits(store, query_text, limit=50)` for a cheap BM25 cut, then re-rank those with the event embedding against claim embeddings.
   - If the FTS table is empty (returns `[]`), call `HybridRetriever().rebuild(store, origins={FOUNDER, VOICE, LITERATURE, SYSTEM})` once, then retry. Log the rebuild — this is a recoverable cold-start, not a failure.
   - Take top-k_claims above `min_score`.
4. **Dedupe across stages:** for each Claim hit, if its embedding cosine against any returned Conclusion hit exceeds 0.85, drop the Claim (it's subsumed; citing the Conclusion is preferred).
5. **Sort** merged list by score descending. Return.

Use `numpy` for cosine; the helper exists from prompt 03.

### 3. Truncation for display

When constructing the `text` field, cap Conclusions at 400 chars and Claims at 300 chars. The UI (prompt 12) will render the full text on-demand; the retrieval hit text is for prompt ingestion and citation-quote selection.

### 4. Failure modes — explicit, not silent

- Empty store → return `[]`, log `retrieval_empty_store`.
- FTS not built → rebuild once, retry, log `retrieval_fts_cold_start_rebuilt`.
- Embedding provider down → re-raise to let caller/scheduler handle backoff.
