# shared-schemas

JSON-Schema definitions for the payloads that flow between the codex
(TypeScript / Prisma) and noosphere (Python / Pydantic) sides of
Theseus. Each schema here is the **single source of truth** for the
wire-form of a cross-language concept; the per-language types are
generated from it.

For the ownership rules (Prisma vs. Pydantic vs. JSON Schema) see
[`docs/architecture/Type_Boundaries.md`](../../docs/architecture/Type_Boundaries.md).

## Layout

```
packages/shared-schemas/
├── conclusion.schema.json
├── mqs.schema.json
├── method_track_record.schema.json
├── revision_event.schema.json
├── citation_verdict.schema.json
├── lineage.schema.json
├── fixtures/
│   └── golden.json                  ← round-tripped by both type-contract tests
└── scripts/
    ├── generate_pydantic.py         ← writes noosphere/noosphere/_generated/
    └── generate_typescript.ts       ← writes theseus-codex/src/lib/_generated/
```

## Regeneration

```bash
# Python target (Pydantic)
python packages/shared-schemas/scripts/generate_pydantic.py

# TypeScript target
tsx packages/shared-schemas/scripts/generate_typescript.ts
```

Both generators write into `_generated/` directories that are
clearly marked and protected by CI lint: each emitted file has an
`AUTOGENERATED` banner and the workflow rejects any file in
`_generated/` that lacks one.

`--check` runs the generator into a temp dir and exits non-zero if the
output differs from the committed tree. CI uses this mode on every PR
that touches a schema, a generator, or any `_generated/` file.

## Toolchain

The generators rely on:

- [`datamodel-code-generator`](https://github.com/koxudaxi/datamodel-code-generator)
  — installed via the root `pyproject.toml` extra `dev` (CI pins
  `0.26.*`). Locally:
  `pip install datamodel-code-generator`.
- [`json-schema-to-typescript`](https://github.com/bcherny/json-schema-to-typescript)
  — installed transiently in CI (`npm install --no-save
  json-schema-to-typescript`). Locally either install it globally or
  rely on `npx`.

No new top-level dependencies are added to either package manifest:
the Pydantic generator is a dev tool, and the TS generator is invoked
through `npx`/`tsx`.

## Runtime validation

The shared schemas are not just a code-generation input — they are
also the spec used at runtime to validate any payload crossing a
language boundary. Codex API handlers validate inbound noosphere
payloads against the schema (`ajv` + `ajv-formats`); the noosphere CLI
validates blobs read out of Prisma columns (`jsonschema`). Validation
failures are routed to the unified founder attention queue under the
`schema-drift` queue rather than silently dropped.

## Adding a new shared concept

1. Author `packages/shared-schemas/<name>.schema.json`. Keep field
   names snake_case to match the Pydantic side; the TS adapters at the
   Prisma boundary handle case translation.
2. Add a golden sample under `fixtures/golden.json` keyed by the
   concept name.
3. Re-run both generators (or commit blank stubs and let CI fail
   you into rerunning — both flows work).
4. Add the concept to the index table in
   `docs/architecture/Type_Boundaries.md`.
5. If the concept also has a Prisma row, leave that row as the
   canonical row type and only use the generated type for wire-form
   payloads (e.g. JSON columns, API bodies, CLI exports).

## Why not Protobuf / Buf / gRPC

We are deliberately not introducing a new RPC framework. The goal is
*alignment of in-process types and ad-hoc JSON payloads*, not a
re-platforming of the wire protocol. JSON Schema fits this shape
because it operates on the JSON we already exchange.
