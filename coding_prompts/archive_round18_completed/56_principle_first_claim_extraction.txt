Implement:

A. Diagnosis. Pull 20 random recent conclusions. For each,
   classify as principle-shaped (transferable, contains a
   rule / criterion / decision-relevant generalisation) vs.
   first-person-shaped ("I think X", "I have since become a
   fan of Y"). Persist counts and 10 worst examples in
   `docs/research/internal/extractor_diagnosis_2026_05_13.md`.

B. New extractor contract. A conclusion's body MUST satisfy:
   - Third-person, generalisable statement of a rule, criterion,
     mechanism, or actionable judgment.
   - Contains at least one of: a quantifiable trigger ("when
     X exceeds Y"), a comparative ("X is more reliable than
     Y when ..."), a counterfactual ("if Z were true, then ..."),
     or a normative rule ("one should ... because ...").
   - Does NOT begin with "I", "we", "my", "our". If the
     source content is purely autobiographical, the extractor
     either extracts the principle implied by the autobiography
     OR refuses to emit a conclusion for that span and logs
     `NO_PRINCIPLE_EXTRACTABLE` with the source span.
   - Cites the source span verbatim (existing constraint —
     preserve it).
   - Carries new structured fields:
     - `principle_kind`: enum
       `RULE | CRITERION | MECHANISM | HEURISTIC | DEFINITION |
        FORMULA | ALGORITHM`
     - `domain_of_applicability`: free-text but bounded to
       300 chars
     - `quantifiable_proxies`: list of metric/data-set names
       that could OPERATIONALISE this principle (feeds prompt 57
       and prompt 64 — the quantitative bridge).
     - `decision_examples`: list of ≤ 3 example decisions this
       principle would inform.

C. System prompt rewrite. The extractor's system prompt lives
   in `noosphere/noosphere/extractors/_prompts/`. Rewrite it
   per the contract above. The new system prompt:
   - Names the contract explicitly.
   - Gives 3 positive examples (first-person source → extracted
     principle) and 3 refusals (first-person source where no
     principle is extractable).
   - Requires the new structured fields in the JSON output.

D. Schema migration. Add the new structured fields to the
   `Conclusion` model (Pydantic + Prisma + SQLModel). Backfill
   plan: existing conclusions get NULL for the new fields and
   a queue entry to be re-extracted (see step F).

E. Re-extraction queue. A founder-confirmable queue at

   conclusion, and the agent's proposed principle-shaped
   replacement. Founder accepts / edits / rejects per row.
