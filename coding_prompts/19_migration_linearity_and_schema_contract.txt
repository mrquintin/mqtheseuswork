Goal: enforce migration integrity across Prisma and Alembic and
the Prisma-vs-SQLModel schema contract. Round 19 added 9+ new
migrations on top of Round 18's stack; without explicit linearity
and contract tests, a single drift produces silent runtime errors
that surface as 500s in production weeks later. This prompt
ships the tests + the pre-commit gate that catches that drift
before it hits sync.

What "linearity" means here:

- Prisma: every migration directory has a strict total order; no
  branching, no skipped sequence numbers; every migration's
  `migration.sql` applies cleanly from a fresh DB to HEAD in
  filename-sorted order.
- Alembic: every revision has exactly one `down_revision` (or
  null for the base); the chain has no branches, no orphans,
  no duplicate revision ids.
- The Prisma migration HEAD and the Alembic HEAD describe the
  same logical schema (modulo Prisma-only Next-specific tables
  and Alembic-only noosphere internals).

What "schema contract" means:

- Every Prisma model that has a SQLModel mirror MUST have:
  - same table name
  - same column names
  - column types compatible (Prisma `String` ↔ SQLModel `str`;
    Prisma `Int` ↔ SQLModel `int`; etc. with documented mapping)
  - same NOT NULL / nullable
  - same indexes (compound key columns + order)
- A documented allow-list of Prisma-only and SQLModel-only
  tables.

Implement:

A. `tests/migration/test_migration_linearity.py` (extend the

   - Lists every Prisma migration directory; asserts the

   - Same checks for Alembic: walks the revision chain from

     orphans; runs `alembic upgrade head`; runs
