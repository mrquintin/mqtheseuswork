# Codex Prompt 2 of 18 — Forecasts data model (Prisma + SQLAlchemy + migrations)

**Sequence:** prompt 2 of 18 (Round 10, Wave A — data foundation). Depends on 01 (FORECASTS_DESIGN.md). Report back with: the Prisma migration filename, the Alembic revision id, and the ER summary printed by `prisma format`.

**Why this prompt exists.** Forecasts owns its own tables; it does not piggyback on `CurrentEvent`/`EventOpinion`. The reason is semantic: a Currents *opinion* is reasoning about something that happened; a Forecasts *prediction* is reasoning about something that will happen, attached to a specific external market and resolved against that market's outcome. Conflating them in shared tables would either bloat columns or force NULLs on every row.

### 1. Prisma additions to `theseus-codex/prisma/schema.prisma`

Add these enums and models (names match FORECASTS_DESIGN.md §3 exactly):

```prisma

model ForecastMarket { ... }

model ForecastBet { ... }
model ForecastPortfolioState { ... }   // unique(organizationId)
model ForecastFollowUpSession { ... }
model ForecastFollowUpMessage { ... }
```

Mandatory columns:
- Every model has `organizationId` + FK to `Organization`, `createdAt`, `updatedAt`.
- `ForecastMarket.externalId` is unique per `(source, externalId)` (compound unique key).
- `ForecastPrediction.marketId` FK to `ForecastMarket`; `@@index([organizationId, status, createdAt])`.
- `ForecastCitation.predictionId` FK to `ForecastPrediction`; on cascade-delete the citation goes with it.
- `ForecastResolution.predictionId` is unique (one resolution per prediction).
- `ForecastBet.predictionId` FK; `mode` defaults to `PAPER`; `liveAuthorizedAt` is `DateTime?` (NULL means "never authorized for live").
- `ForecastPortfolioState` is `@@unique([organizationId])`. Tracks `paperBalanceUsd`, `liveBalanceUsd`, `dailyLossUsd`, `dailyLossWindowResetAt`, `killSwitchEngaged Boolean @default(false)`, `killSwitchReason String?`.
- Foreign keys to `Organization` add the new relation backrefs (mirror what was done for `currentEvents`).

Run `npx prisma format` and then generate a migration named `<ts>_forecasts_data_model` via `npx prisma migrate dev --create-only` (do not auto-apply; the migration file is committed but applied by the operator).
