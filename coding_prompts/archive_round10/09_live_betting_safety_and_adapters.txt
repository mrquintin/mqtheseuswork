# Codex Prompt 9 of 18 — Live trading safety layer + Polymarket/Kalshi adapters (DEFAULT OFF)

**Sequence:** prompt 9 of 18 (Round 10, Wave C — bets, live mode behind gates). Depends on 02, 08. Report back with: every gate that must be open before a live bet submits, the kill-switch reason codes, and proof that running the test suite NEVER hits a real exchange (httpx-mock or respx assertion count).

**Why this prompt exists.** Live-money trading is a footgun. The voice memo asks for it and we will ship the capability, but only behind a multi-layered consent fence. This prompt builds the safety layer first, then the two adapters that consume it. The default deployment posture is: live trading is OFF, the env flag is not set, no credentials are present, and the code paths that would call an exchange short-circuit on the first gate.

**SAFETY CONTRACT (read this before writing anything).**
The system you are building must satisfy ALL of the following at runtime, on every live-trade submission attempt, in order. ANY gate failing → no submit, structured-log line emitted, kill-switch *not* engaged (this is a normal control-flow case, not an emergency).

1. `FORECASTS_LIVE_TRADING_ENABLED == "true"` (string compare, lowercased).
2. The exchange adapter's credentials are configured (Polymarket: a private key; Kalshi: an API key id + private key PEM).
3. The relevant prediction has `live_authorized_at != NULL` — set only via the operator endpoint added in prompt 11.
4. The bet has a per-bet operator confirmation: status transition AUTHORIZED → CONFIRMED via a CSRF-protected POST handled by the operator UI in prompt 14, NOT by automation.
5. `stake_usd <= FORECASTS_MAX_STAKE_USD`.
6. The day's running loss does not exceed `FORECASTS_MAX_DAILY_LOSS_USD` (read from `ForecastPortfolioState.dailyLossUsd`).
7. `ForecastPortfolioState.killSwitchEngaged == False`.
8. The portfolio has sufficient `liveBalanceUsd` for the stake (no leverage, no margin, no shorts beyond `NO`-side native exposure).

The bet only submits when 1–8 all hold. There are no exceptions, no operator overrides, no "force" flags. To bypass any gate you must edit the code — and the code review for that edit is a separate decision.

**Before you start — read current state:**
1. Read `noosphere/noosphere/forecasts/paper_bet_engine.py` from prompt 08. Live engine reuses the staking math (Kelly, edge threshold) — it diverges only at the *submission* step.
2. Read `noosphere/noosphere/forecasts/_kalshi_client.py` and `_polymarket_client.py`. Live adapters extend these clients with order-placement methods.
3. Read `coding_prompts/FORECASTS_DESIGN.md` §6 for the mode state machine.

Auto-engagement triggers (each is a separate detection function in `safety.py`):
- Daily loss exceeds `FORECASTS_KILL_SWITCH_AUTO_THRESHOLD_USD` → reason `DAILY_LOSS_THRESHOLD`.
- Three consecutive bet submissions return exchange errors → reason `EXCHANGE_ERROR_STREAK`.
- Calibration over the last 50 resolved predictions degrades to mean Brier > 0.30 → reason `CALIBRATION_DEGRADED`.

Disengagement requires an operator-supplied `note ≥ 20 chars` and is logged.
