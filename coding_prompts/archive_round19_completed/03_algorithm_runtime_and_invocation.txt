Goal: the runtime that takes ACTIVE algorithms and FIRES them
against the world. The runtime observes incoming inputs from
Currents (live X feed), market data, uploaded artifacts, and
manual operator entries, evaluates each algorithm's trigger
predicate, and when it matches, runs the reasoning chain to
produce an AlgorithmInvocation with derivedOutput +
reasoningTrace.

This is where the founder's vision becomes operational:
algorithms fire on real-world events, produce predictions
with reasoning attached, and queue bet recommendations for the
portfolio agent.

What runtime IS — pipeline:

```
Currents events ─┐
Market data ─────┼─→ InputResolver ──→ Observation
Artifacts ───────┤       │
Operator inputs ─┘       │
                         ▼
                   For each ACTIVE algorithm:
                     evaluate trigger_predicate
                     ├─ False → skip
                     └─ True  → run reasoning_chain
                                ├─ DETECT step → record observed inputs
                                ├─ APPLY_PRINCIPLE step → invoke principle,
                                │                          record derived fact
                                ├─ SYNTHESIZE step → combine derived facts
                                │                     into output structure
                                └─ OUTPUT step → emit derivedOutput
                                  + confidence_low / high
                                  + reasoning_trace (human-readable)
                                  → persist AlgorithmInvocation
```

What it MUST NOT do:
- Fire the same algorithm twice on the same observation
  (idempotency).
- Auto-execute a bet. The invocation can suggest a bet (in
  `betImplied`) but actual order placement happens later in the
  portfolio-agent prompt (10).
- Modify the algorithm. Algorithms only change via the triage UI
  (prompt 02).

Implement:

A. `noosphere/noosphere/algorithms/input_resolver.py`:
   - `class InputResolver`:
     - `async resolve(input_spec) -> InputObservation | None`:
       takes an `AlgorithmInput` spec and returns the most-recent
       valid observation, sourced from the input's
       `observability_source`.
     - Supported sources (extend over time):
       - `currents.x.<field>` — pulls from the latest CurrentEvent
         matching a filter. `<field>` extracts a specific
         derived metric (e.g., `escalation_index`,
         `mentions_per_day`, `bilateral_sentiment_score`).
       - `markets.polymarket.<condition_id>.<field>` — Polymarket
         live state.
       - `markets.kalshi.<ticker>.<field>` — Kalshi live state.
       - `markets.alpaca.<symbol>.<field>` — equity price /
         volume / etc.
       - `artifact.field.<artifact_id>.<field>` — derived from
         an uploaded artifact (e.g., the founder enters a
         spreadsheet of bilateral spending; cells are addressable).
       - `manual.operator.<key>` — operator-entered values via
         the algorithm-input panel (prompt 04 builds this).
     - Returns None if the source cannot be observed right now;
       caller decides what to do with the absence.
   - Pluggable adapter registry so new sources can be added by
     other prompts.

B. `noosphere/noosphere/algorithms/runtime.py`:
   - `class AlgorithmRuntime`:
     - `async tick_once(store, *, now) -> TickResult`:
       1. List all ACTIVE algorithms for the org.
       2. For each algorithm: resolve every input via InputResolver.
       3. If any required input is None → skip with reason
          `INPUT_UNAVAILABLE`.
       4. Evaluate the trigger predicate in a sandboxed
          environment (use `asteval` or a hand-rolled AST walker
          allowing only ops the validator (prompt 01) accepts).
       5. If False → skip.
       6. If True → execute the reasoning chain step-by-step.
          - DETECT: record the observation.
          - APPLY_PRINCIPLE: load the principle, append a
            "Applied principle '...' → derived fact '...'" trace
            line. The derived fact is generated by a brief Haiku
            call grounded in the principle text + the observed
            inputs. The call must abstain rather than fabricate.
          - SYNTHESIZE: combine derived facts into the structured
            output schema. Done by a single Haiku call with the
            algorithm's output schema as JSON-schema constraint.
          - OUTPUT: persist the AlgorithmInvocation with full trace.
       7. Idempotency: an invocation has a deterministic key
          `(algorithm_id, sha256(canonical_input_observation_json))`.
          If a row with that key already exists in the last
          `IDEMPOTENCY_WINDOW_SECONDS` (default 600), skip.
       8. Emit a structured-log line per invocation:
          algorithm name, trigger inputs (sanitised), output
          headline.
   - `class TickResult`:
       fired: int
       skipped_no_input: int
       skipped_predicate_false: int
       skipped_idempotent: int
       errors: list[str]

C. Scheduler integration. The Forecasts scheduler from Round 10
   prompt 19 (now reorganised by Round 18) gains a new sub-loop:
   `algorithms_tick` with default interval 60s (configurable via
   `ALGORITHMS_TICK_INTERVAL_S`). The sub-loop calls
   `runtime.tick_once`. Budget guard reuses the prompt-02 budget.

D. Resolution sub-loop. A separate `algorithms_resolution_tick`
   sub-loop runs every 30 min and, for each unresolved
   invocation whose `predictedHorizon` has elapsed:
   1. Resolve the actual outcome via the same observability_source
      machinery (the world has moved on; ask it).
   2. Compute correctness + brierEquivalent for probabilistic
      outputs.
   3. Persist resolution.
   4. Fire a structured-log event so the calibration tracker
      (prompt 05) picks it up.

E. CLI:
   - `noosphere algorithms tick --once` — run a single tick.
   - `noosphere algorithms run` — standing loop (used by the
     scheduler container in production).
   - `noosphere algorithms fire --algorithm <id> --inputs
      file.json` — force a fire with specific inputs (for
      debugging; sets a `forced=true` flag on the invocation so
      they're filterable in the UI).

F. Tests:
   - Trigger-true happy path: a synthetic ACTIVE algorithm modeled
     after the arms-race example, all inputs resolvable from fixture
     adapters, fires once. The reasoning trace is recorded.
   - Trigger-false: predicate evaluates False, no invocation.
   - Input-unavailable: one input is None → skip with the right
     reason; the algorithm is not penalised.
   - Idempotency: tick twice with identical input → one row.
   - Sandbox: an adversarial trigger predicate (eval-bait string
     somehow promoted) → runtime refuses and emits a
     structured-log warning; algorithm is auto-paused after 3
     refusals.
   - Resolution path: a synthetic invocation past its horizon
     resolves cleanly against a controlled "actual outcome"
     fixture.

Constraints
- The runtime NEVER places a bet. It produces an invocation that
  describes the bet the algorithm implies; the portfolio agent
  (prompt 12) decides whether to act on it.
- The reasoning trace is the audit artifact. Every step is
  human-readable, every principle invoked is named.
- A single algorithm cannot consume more than
  `ALGORITHMS_MAX_TOKENS_PER_FIRE` tokens (default 6000). Over →
  abstain with reason.
- The runtime is robust to a single algorithm's failure: an error
  in algorithm A's reasoning chain does not stop B's tick.
