Goal: when Noosphere ingests a new conclusion or claim, do not test its
coherence against every conclusion in the corpus. As the corpus grows that
is O(n) per ingest and quadratic over time. Instead, locate the *domain
neighborhood* of the new idea via approximate nearest-neighbor (ANN) search
in the embedding space and run coherence/contradiction tests only against
the local cluster, plus a small random sample from outside the cluster as a
safety check.

Two concrete failure modes to fix:
- Today's coherence pipeline either iterates the corpus or batches
  arbitrarily. There is no formal notion of "the local domain of an idea."
- The contradiction surface explodes once the corpus passes a few thousand
  conclusions because every layer of the six-layer engine runs on the full
  context.

A. A locality index module.

     ```
     class DomainLocalityIndex:
         def upsert(self, proposition_id: str, embedding: np.ndarray) -> None
         def remove(self, proposition_id: str) -> None
         def neighbors(
             self,
             query_embedding: np.ndarray,
             *,
             k: int = 64,
             radius: float | None = None,
             include_outside_sample: int = 8,
         ) -> NeighborResult
     ```
     `NeighborResult` distinguishes `local_ids` (top-k or in-radius) from
     `outside_sample_ids` (a small random sample drawn from the rest of the
     corpus to keep the test honest about distant contradictions).
   - The implementation must be ANN-based, not brute-force. Prefer hnswlib
     (already permitted in the project ‚Äî verify by reading
     `noosphere/pyproject.toml` and `requirements.txt`); fall back to a

     safety net. Document the fallback clearly. If hnswlib is not yet a
     dependency, add it to `noosphere/pyproject.toml` with a defended pin.
   - Persistence: serialize the ANN index to
     `noosphere_data/coherence/locality.bin` and track the contained id set
     alongside it as JSON for cheap consistency checks. Index loads
     lazily; if the file is missing the index reconstructs from the
     existing embedding column on first use and writes itself out.
   - Idempotency: `upsert` is safe to call repeatedly with the same id and
     the same vector (no duplicate entries; vector replacement is allowed).
