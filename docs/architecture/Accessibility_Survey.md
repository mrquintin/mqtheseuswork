# Accessibility Survey

**Standard:** WCAG 2.1 AA
**Scope:** every public, unauthenticated surface of the Theseus codex
plus the founder-portal chrome that public readers can reach during
sign-in (login, account creation, password reset).
**Methodology:** automated axe-core sweep + manual checklist, repeated
nightly in CI (`.github/workflows/a11y_nightly.yml`). The latest
automated sweep is regenerated by that workflow; this document is the
human-curated companion that records findings, their disposition, and
the manual checks the scanner cannot perform.

The firm publishes for serious readers. Serious readers include people
who use screen readers, keyboard-only navigation, 200% text zoom,
reduced-motion preferences, and high-contrast modes. None of those are
edge cases — they are the readership.

## 3. Manual checklist

The scanner cannot evaluate the items below. Every page was driven
once with keyboard only, once with VoiceOver on macOS, and once with
200% text zoom.

### 3.1 Keyboard

- [x] Tab reaches every interactive element in the chrome (skip-link,
      logo, nav links, theme toggle, CTA, footer links). Tested on `/`,
      `/c/[slug]`, `/currents`, `/forecasts`.
- [x] Tab order matches visual reading order (top-to-bottom,
      left-to-right) on each surveyed page.
- [x] The mobile drawer trap-focuses correctly. Tab cycles, Shift-Tab
      cycles back, Esc closes and restores focus to the hamburger
      trigger.
- [x] The command palette now traps focus while open (Tab/Shift-Tab
      cycle inside the dialog). Previously focus could escape into the
      page underneath while the palette was visible — repaired in this
      pass.
- [x] Citation popover already trapped focus and restored on close;
      verified.
- [x] No focusable element is keyboard-unreachable due to `tabindex`,
      `pointer-events`, or off-screen positioning.

### 3.2 Focus visibility

- [x] Every interactive element has a visible `:focus-visible` style.
      The default ring is a 2px `--amber` outline with 2px inset glow.
      Body links use a 1px underline when focused.
- [x] Skip-to-content link is visually hidden until focused, then
      slides to the top-left at `top: 0.75rem` with the amber phosphor
      background. Verified in light and dark themes.

### 3.3 Images and decorative elements

- [x] AsciiHero and AsciiRuin are presentational (figure with
      `aria-hidden`). The accompanying caption carries the meaning.
- [x] Decorative pulses (`CurrentsNavPulse`, `ForecastsNavPulse`) use
      `aria-hidden` on the dot; the accessible name is on the parent
      link.
- [x] Provenance heatmap, calibration plots, cascade trees have
      `<figcaption>` or `aria-label` describing the chart contents.
      Underlying data is also rendered as a table on each page so
      screen-reader users can read the values directly.

### 3.4 Color is not the only signal

- [x] Calibration bars: numeric percent is always rendered alongside
      the bar fill. The bar has a high-contrast border so the fill
      level is shape-visible.
- [x] Provenance shading: each sentence carries a `data-tier` attribute
      that maps to a textual label exposed in the popover. Color is
      reinforcement, not the only signal.
- [x] Status pills (`Pill` primitive): every variant carries text. The
      `tone` palette ships with both fg/bg pairs so the label is
      readable on the colored chip.
- [x] Source-standing pills (retracted/corrected/disputed/expired)
      always include the standing word.

### 3.5 Live regions

- [x] Currents SSE feed announces new entries via
      `aria-live="polite"` on the feed container.
- [x] Forecast probability updates use `aria-live="polite"` on the
      changed cell. Updates are throttled to once per 2s so the
      announcement queue doesn't flood.
- [x] Toast notifications (success/error) use `aria-live="assertive"`
      and `role="status"`.

### 3.6 Skip link

- [x] Present on every public page via the root layout. Renders as the
      first element of `<body>`. Focusable, target is `#main-content`,
      target is itself focusable (`tabindex="-1"`).

### 3.7 Heading hierarchy

skipped levels, headings reflect document structure.

## 4. Color tokens and contrast

The Amber-Oracle palette is preserved. The two failing tokens were
both dim foreground tones that did not survive the 4.5:1 threshold at
the smallest body sizes.

| Token              | Theme | Before  | After    | Contrast on `--stone` | Notes |
| ------------------ | ----- | ------- | -------- | --------------------- | ----- |
| `--parchment-dim`  | light | `#6b5e48` | `#5a4e3a` | 5.06:1 (was 3.92:1)   | Used for `.public-muted`. Darkening is barely perceptible (ΔE ≈ 4) and preserves the aged-parchment feel. |
| `--amber-deep`     | dark  | `#5e4617` | unchanged | 4.7:1 large only      | Marked for review — currently only used as a disabled border, never as text. Body-text use is prohibited via lint. |

All other tokens already met the threshold:

- `--amber` (`#e9a338`) on `--stone` (`#08070a`): 9.8:1
- `--parchment` (`#efe2c7`) on `--stone`: 14.7:1
- `--parchment-dim` (`#9c8f72`) on `--stone` (dark): 5.4:1
- `--success` (`#7ea83a`) on `--stone`: 7.4:1
- `--info` (`#3a7aa8`) on `--stone`: 4.6:1
- `--ember` (`#c94a1f`) on `--stone`: 4.8:1

The `tokens.ts` export now records the audited ratios as inline
comments next to each token, so future palette tweaks have to
re-justify the change.

## 5. Reduced-motion variants

Every animation added in Round 17–18 now has a reduced-motion
variant. The rule is: under `prefers-reduced-motion: reduce`, an
animation either disables entirely or collapses to its final
(steady) state. Animations are never load-bearing — they reinforce
information that is already present in text or shape.

| Animation               | Reduced-motion behavior |
| ----------------------- | ----------------------- |
| `.currents-pulse`       | `animation: none`; dot remains visible at full opacity. |
| `.currents-fade-in`     | `animation: none`; element appears in final state. |
| `.forecasts-prob-shimmer` | `animation: none`; bar remains at steady opacity. |
| `.forecasts-prob-bar .fill` | `transition: none`; width snaps. |
| `.gate-*`               | Already disabled (Round 14 baseline). |
| `.crt-fade-in`          | Already disabled. |
| AsciiHero shimmer       | Component-side `prefers-reduced-motion` check disables the rAF loop. |
| Nav transitions         | Component-side check; instant transitions when reduce is set. |
| Explorer pan            | Component-side check; pan disabled, jump-to-target instead. |

CRT scan-line overlay also short-circuits under reduced motion (see
`CRTOverlay.tsx`).

## 6. Process

- Findings table above is the ground truth. CI nightly regenerates
  the automated section; the manual section is updated by hand when
  a new public surface ships.
- Failures in CI block the next deploy. The PR comment lists the
  failing rule, page, and offending node so the fix can be local.
- A11y review is owned by whoever ships the page. This file is a
  shared artifact, not a hand-off doc.
