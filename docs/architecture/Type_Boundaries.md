# Type Boundaries

This doc records who owns the type for each shared concept that flows
between the three layers of Theseus:

| Layer        | Language   | Surface                                        |
| ------------ | ---------- | ---------------------------------------------- |
| Database     | TypeScript | `theseus-codex/prisma/schema.prisma` (Prisma)  |
| Codex        | TypeScript | `theseus-codex/src/**/*.ts`                    |
| Noosphere    | Python     | `noosphere/noosphere/**/*.py` (Pydantic)       |

Before Round 18 each concept tended to have its definition repeated on
two or three sides, with `noosphere → codex` JSON paths quietly
diverging from `Prisma → codex` row shapes. This file is the index for
which side owns what, and what generation flow keeps the others honest.

## Ownership rules

1. **Database-backed entities** are owned by Prisma. The TS row type is
   `Prisma.<Model>`; the codex never hand-writes a parallel interface.
   Noosphere does not query Prisma directly — when noosphere needs to
   reason about a DB-backed entity it uses the corresponding generated
   Pydantic model (case (3) below).
2. **Computational primitives** that exist only inside noosphere (e.g.
   cascade-graph internals, NLI judge intermediates) are owned by
   Pydantic. They never cross a wire and never appear in the codex.
3. **Shared API payloads** that *do* cross a wire — noosphere ↔ codex
   over an HTTP API, a CLI export, or a row that is read on both sides —
   are owned by JSON Schema, with both Pydantic and TypeScript generated
   from it. Manual edits to either generated side are blocked by CI.

### Why JSON Schema and not a single ORM

We considered making Prisma the single owner and generating Pydantic
from `schema.prisma`. Two problems made that unworkable:

- Noosphere stores objects that have no Prisma row (e.g. `MQS` lives in
  Python, with codex only persisting a pruned view). A schema-first
  artefact is the natural fit for "the JSON shape that both sides
  agree on", which is not always 1:1 with a table.
- Prisma's generator emits TypeScript only. Generating Python from
  Prisma would require a custom generator and a Prisma version pin we
  don't want to take on.

## Generation flow

```
packages/shared-schemas/<name>.schema.json
        │
        ├── scripts/generate_pydantic.py    ──▶ noosphere/noosphere/_generated/<name>.py
        │
        └── scripts/generate_typescript.ts  ──▶ theseus-codex/src/lib/_generated/<name>.ts
```

Generated trees are committed (so editors have type completion without
running the generators) and protected by `.github/workflows/type-contracts.yml`:

- Re-run both generators in a temp dir.
- `diff` against the committed `_generated/` trees.
- Lint: every file in `_generated/` must contain `AUTOGENERATED` in its
  first 200 bytes.
- Run the type-contract tests on both sides against the golden fixture.

Drift between the schema and any generated tree fails the build.

## Runtime validation at boundaries

At every cross-language boundary we validate against the JSON Schema
rather than trusting the producer:

- **Noosphere → codex (HTTP)**: codex API handlers that consume a
  noosphere payload run `validateAgainstSchema(name, payload)` before
  trusting the body. Failures log to the unified attention queue (Round
  17 prompt 34) under the `schema-drift` queue with the offending
  schema name, the validator error string, and a redacted payload
  excerpt. They do not throw silently or fall back to "best-effort"
  parsing.
- **Codex → noosphere (DB row)**: noosphere readers that interpret JSON
  blobs out of Prisma columns (`RevisionEvent.planJson`,
  `CitationVerdict.raw_payload`, etc.) parse them through the
  generated Pydantic model rather than `dict.get(...)`. A failed parse
  is surfaced to the attention queue via the same mechanism on the
  noosphere side.

In both cases the failure mode is "make the drift visible to the
founder," never "drop the row."

## Migrating duplicates

Round 17 left several parallel definitions for the same concept on the
TS and Python sides. The standard migration step is:

1. Write or update the JSON Schema in `packages/shared-schemas/`.
2. Regenerate both `_generated/` trees.
3. Replace the hand-written types on the noosphere side with
   `from noosphere._generated import X`.
4. Replace the hand-written TS interfaces with imports from
   `@/lib/_generated`.
5. For DB-backed concepts whose codex-side type was a hand-rolled
   interface, replace it with `Prisma.<Model>` directly — those types
   do not need to go through the JSON Schema bridge.
6. Add the concept to the index table in this doc.

## Non-goals

- We are not introducing a new RPC framework (gRPC, Buf, etc.). The
  point of this work is *alignment*, not re-platforming.
- We are not auto-generating Prisma models from JSON Schema. Prisma's
  TS output already covers the DB side; duplicating the row shape into
  the JSON Schema bridge would invite a different kind of drift.
